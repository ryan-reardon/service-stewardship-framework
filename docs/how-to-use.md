# How to Use the Service Stewardship Framework

The Service Stewardship Framework is not intended to be a static document.

It is an operating instrument.

This guide explains how to apply the framework and domain modules in practice to drive structured service ownership, measurable maturity progression, and intentional leadership.

---

## Purpose of the Domain Modules

Each of the 12 domain modules exists to:

- Make implicit ownership expectations explicit
- Provide structured evaluation criteria
- Identify systemic risk exposure
- Guide improvement planning
- Establish governance rhythm

The modules are designed to be used actively, not archived.

---

# Operating Model

The framework supports three primary use cases:

1. Structured Self-Assessment  
2. Leadership Governance Reviews  
3. Targeted Improvement Planning  

---

## 1. Structured Self-Assessment

Start with the consolidated maturity questionnaire.

### Step 1: Baseline Scoring

- Evaluate all 12 domains.
- Score each domain using the maturity model (Reactive → Proactive).
- Provide evidence for each score.
- Avoid aspirational scoring.

The goal is clarity, not perfection.

### Step 2: Identify Priority Domains

After scoring:

- Identify the 1–3 lowest maturity domains.
- Identify any domains with high operational or financial risk.
- Prioritize intentionally.

Do not attempt to elevate all domains simultaneously.

### Step 3: Deep-Dive with Domain Modules

Open the corresponding domain module in `/modules`and:

- Evaluate each section.
- Confirm existence of required artifacts.
- Review metrics and governance cadence.
- Identify structural gaps.

This prevents overconfidence and ensures evidence-based assessment.

---

## 2. Leadership Governance Reviews

The framework is designed to support structured leadership cadence.

### Recommended Review Rhythm

- Quarterly: Review 2–3 domains in depth  
- Annually: Full 12-domain reassessment  

### Domain Review Agenda

For each domain review:

1. Present current maturity level  
2. Present supporting metrics  
3. Identify risk exposure  
4. Highlight structural gaps  
5. Define improvement priorities  

The goal is not reporting.  
The goal is decision-making.

---

## 3. Improvement Planning

When a domain scores below target maturity:

### Convert Gaps into Action

- Translate gaps into backlog items.
- Assign accountable owner.
- Define target maturity level.
- Define measurable outcome.
- Set review checkpoint.

### Focus on Structural Improvements

Improvements should prioritize:

- Governance clarity
- Artifact creation
- Metric instrumentation
- Risk reduction
- Sustainable workload balance

Avoid cosmetic documentation updates without operational change.

---

# Maturity Progression Approach

The framework uses four maturity levels:

- **Level 1: Reactive**
- **Level 2: Managed**
- **Level 3: Optimized**
- **Level 4: Proactive**

Progression should be staged.

Do not attempt to move directly from Level 1 to Level 4.

Each maturity transition should correspond to:

- Increased visibility
- Improved governance
- Reduced risk exposure
- Better decision clarity

---

# Suggested Adoption Plan for Engineering Managers

### Month 1
- Complete full 12-domain baseline assessment.
- Validate scoring with stakeholders.
- Identify top 2 priority domains.

### Months 2–3
- Deep dive into priority domains.
- Implement structural improvements.
- Establish governance cadence.

### Quarterly
- Reassess impacted domains.
- Track maturity movement.
- Adjust roadmap allocation as needed.

### Annually
- Complete full reassessment.
- Recalibrate strategic posture.
- Update investment thesis if required.

---

# Common Misuse Patterns

Avoid using the framework as:

- A compliance checklist
- A one-time audit exercise
- A bureaucratic reporting layer
- A maturity score competition

If the framework does not influence decisions, trade-offs, or investment allocation, it is not being used correctly.

---

# Core Principle

Services do not drift into excellence.\
They drift into entropy.

The domain modules exist to ensure that leadership is intentional, structured, and multidimensional across:

- Strategy  
- Economics  
- Architecture  
- Reliability  
- Security  
- Organizational health  
- Portfolio alignment  

If a service cannot be described clearly across these dimensions, stewardship is incomplete.

---

# Closing Guidance

The Service Stewardship Framework is designed to evolve.\
As services scale and organizations change, stewardship practices must adapt.\
Treat the framework as a living operating model.

Refine it.
Challenge it.
Improve it.

Intentional stewardship is what sustains durable technology services.
